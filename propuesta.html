Documentacion Tecnica del Sistema Web

Proyecto de Tesis - Deteccion de Retinopatia Diabetica

1. Introduccion

Este documento presenta la arquitectura tecnica del sistema web desarrollado para el proyecto de tesis de deteccion de Retinopatia Diabetica mediante inteligencia artificial. El sistema permite a usuarios cargar imagenes de fondo de ojo y obtener un analisis automatizado del nivel de severidad de la enfermedad.

La plataforma web fue desarrollada utilizando tecnologias modernas que garantizan escalabilidad, rendimiento y facilidad de uso, tanto para pacientes como para profesionales medicos.

2. Interfaz de Usuario (Frontend)

La interfaz visual del sistema esta construida con tecnologias web modernas que garantizan una experiencia fluida, responsiva y accesible desde cualquier dispositivo (computadoras, tablets, smartphones).

React 19

Libreria de JavaScript desarrollada por Meta (Facebook) para construir interfaces de usuario interactivas y dinamicas. Permite crear componentes reutilizables y manejar el estado de la aplicacion de forma eficiente, resultando en una experiencia de usuario fluida similar a aplicaciones nativas.

Tailwind CSS

Framework de estilos CSS que permite diseñar interfaces modernas de forma rapida y consistente mediante clases utilitarias. Garantiza que todos los elementos visuales mantengan un diseño coherente en toda la aplicacion.

Biblioteca de Componentes (Radix UI + shadcn/ui)

Conjunto de componentes de interfaz pre-construidos (botones, formularios, dialogos, tablas) que cumplen con estandares de accesibilidad web (WCAG). Esto asegura que el sistema sea utilizable por personas con diferentes capacidades y desde distintos dispositivos.

React Router

Sistema de navegacion que permite crear una aplicacion de pagina unica (Single Page Application). El usuario puede navegar entre diferentes secciones sin que la pagina se recargue completamente, mejorando significativamente la velocidad de respuesta.

3. Servidor y API (Backend)

El backend es el nucleo logico del sistema. Procesa las peticiones del frontend, gestiona el acceso a la base de datos y coordina la comunicacion con el modelo de inteligencia artificial.

FastAPI (Python)

Framework web moderno para Python que permite crear APIs REST de alto rendimiento. FastAPI fue seleccionado por su velocidad (comparable a frameworks en NodeJS y Go), su capacidad para validar datos automaticamente y su excelente integracion con bibliotecas de Machine Learning como TensorFlow, PyTorch y scikit-learn.

API REST (Comunicacion Frontend-Backend)

El frontend se comunica con el backend mediante peticiones HTTP siguiendo el estandar REST (Representational State Transfer). Los datos se intercambian en formato JSON, un formato ligero y universalmente soportado. Esta arquitectura permite separar completamente la interfaz visual del procesamiento de datos.

Motor de Asincronismo

FastAPI utiliza programacion asincrona (async/await) que permite procesar multiples peticiones simultaneamente sin bloquear el servidor. Esto es especialmente importante cuando el modelo de IA esta procesando una imagen, ya que otros usuarios pueden seguir utilizando el sistema sin experimentar demoras.

4. Base de Datos

MongoDB

Base de datos NoSQL orientada a documentos. A diferencia de bases de datos relacionales tradicionales (SQL), MongoDB almacena informacion en formato JSON flexible, lo que facilita almacenar datos semi-estructurados como contenido de paginas web, imagenes y resultados de analisis.

La base de datos almacena: contenido editable de las paginas del sitio web, metadatos de imagenes analizadas, y opcionalmente un historial de predicciones para analisis estadistico posterior.

5. Modelo de Inteligencia Artificial

El componente central del sistema es un modelo de Deep Learning (aprendizaje profundo) entrenado especificamente para detectar y clasificar Retinopatia Diabetica en imagenes de fondo de ojo.

5.1 Flujo de Procesamiento

El proceso de analisis sigue los siguientes pasos:

  1. El usuario carga una imagen del fondo de ojo desde la interfaz web
  2. El frontend envia la imagen al backend mediante una peticion HTTP POST
  3. El backend valida el formato de la imagen y la envia al modelo de IA
  4. El modelo preprocesa la imagen (redimensionamiento, normalizacion)
  5. El modelo procesa la imagen a traves de su red neuronal
  6. Se genera una prediccion con el nivel de severidad y un porcentaje de confianza
  7. El resultado se envia de vuelta al frontend
  8. El usuario visualiza el resultado con recomendaciones clinicas

5.2 Clasificacion de Severidad

El modelo clasifica las imagenes en 5 categorias segun la escala clinica estandar:

  • Sin Retinopatia: No se detectan signos de la enfermedad
  • Retinopatia Diabetica Leve: Presencia minima de microaneurismas
  • Retinopatia Diabetica Moderada: Multiples microaneurismas y hemorragias
  • Retinopatia Diabetica Severa: Hemorragias extensas y anomalias vasculares
  • Retinopatia Diabetica Proliferativa: Neovascularizacion y riesgo alto de perdida visual

Especificaciones Tecnicas del Modelo (A Completar)

Esta seccion sera completada con los detalles especificos del modelo entrenado:

  • Arquitectura de red neuronal utilizada (ResNet, VGG, EfficientNet, DenseNet, etc.)
  • Dataset utilizado para entrenamiento (nombre, numero de imagenes, distribucion de clases)
  • Metricas de rendimiento del modelo:
    • Accuracy (precision global)
    • Precision, Recall y F1-Score por clase
    • Matriz de confusion
    • AUC-ROC
  • Tecnicas de preprocesamiento de imagenes aplicadas
  • Tecnicas de data augmentation utilizadas durante entrenamiento
  • Hiperparametros del modelo (learning rate, batch size, epocas, optimizer)
  • Umbrales de confianza para clasificacion
  • Tiempo promedio de inferencia por imagen

6. Infraestructura y Arquitectura de Deployment

El sistema esta desplegado en Railway, una plataforma cloud que permite deployment automatico desde repositorios Git. La arquitectura sigue un patron de microservicios donde cada componente funciona de forma independiente.

6.1 Arquitectura de Servicios

Servicio 1: Aplicacion Web (Frontend + Backend)

Contenedor Docker que incluye la aplicacion React compilada y el servidor FastAPI ejecutandose en un solo servicio. El backend sirve los archivos estaticos del frontend y expone la API REST. Esta consolidacion simplifica el deployment y reduce costos operativos sin sacrificar rendimiento.

Servicio 2: Base de Datos MongoDB

Instancia independiente de MongoDB ejecutandose en un contenedor separado. La separacion de la base de datos permite:

  • Escalamiento independiente de recursos computacionales
  • Implementacion de backups automaticos sin afectar la aplicacion
  • Configuracion de replicas para alta disponibilidad
  • Persistencia de datos incluso si se actualiza la aplicacion

Servicio 3: Microservicio de Inteligencia Artificial

Servicio independiente dedicado exclusivamente al modelo de Deep Learning. Esta separacion arquitectonica permite:

  • Asignacion de recursos computacionales especificos (CPU/GPU optimizada para ML)
  • Escalamiento horizontal (multiples instancias del modelo en paralelo)
  • Actualizacion del modelo sin reiniciar la aplicacion web
  • Monitoreo independiente del rendimiento del modelo
  • Posibilidad de implementar A/B testing entre versiones del modelo

6.2 Justificacion de la Arquitectura Seleccionada

Separacion de Servicios

La arquitectura de microservicios permite que cada componente del sistema tenga responsabilidades especificas y bien definidas. Esto facilita el mantenimiento a largo plazo y permite que diferentes equipos trabajen en componentes distintos sin interferencias.

Escalabilidad Independiente

En escenarios de alta demanda, es probable que el modelo de IA requiera mas recursos computacionales que el frontend o la base de datos. La arquitectura permite escalar unicamente el componente que lo necesite, optimizando costos. Por ejemplo, se pueden añadir multiples instancias del servicio de IA con GPUs mientras el frontend mantiene sus recursos actuales.

Alta Disponibilidad y Resiliencia

Si un servicio presenta fallas, los demas pueden continuar operando. Por ejemplo, si el modelo de IA requiere actualizacion o mantenimiento, el sitio web puede seguir funcionando y mostrar un mensaje apropiado, en lugar de que toda la aplicacion quede inaccesible.

Optimizacion de Costos

Railway cobra segun el uso real de recursos (CPU, memoria, ancho de banda). Al tener servicios separados, se puede:

  • Asignar recursos costosos (GPU) solo al servicio de IA
  • Escalar automaticamente segun demanda (auto-scaling)
  • Monitorear y optimizar el consumo de cada servicio independientemente

Rendimiento del Modelo de IA

El procesamiento de imagenes con redes neuronales profundas es computacionalmente intensivo. Al tener un microservicio dedicado, se puede:

  • Utilizar hardware especializado (GPUs NVIDIA para inferencia rapida)
  • Implementar colas de procesamiento para manejar multiples peticiones
  • Cachear resultados de imagenes procesadas previamente
  • No afectar el rendimiento del frontend durante procesos intensivos

Deployment Continuo

Railway se integra con repositorios Git (GitHub, GitLab) para realizar deployment automatico cada vez que se actualiza el codigo. Esto permite:

  • Actualizaciones rapidas sin downtime
  • Rollback automatico en caso de errores
  • Testing automatizado antes de deployment a produccion

7. Justificacion de las Tecnologias Seleccionadas

Criterios de Seleccion Tecnica

React + Tailwind CSS

React es la biblioteca de JavaScript mas utilizada en la industria (segun Stack Overflow Survey 2024), con una comunidad activa de millones de desarrolladores, documentacion extensa y ecosistema maduro. Tailwind CSS permite desarrollo rapido sin requerir experiencia profunda en CSS tradicional.

FastAPI (Python)

Python es el lenguaje dominante en ciencia de datos y Machine Learning. FastAPI fue seleccionado por ser el framework web mas rapido de Python (comparable a NodeJS y Go en benchmarks), con validacion automatica de datos mediante Pydantic y generacion automatica de documentacion interactiva (Swagger/OpenAPI). Su sintaxis moderna con async/await permite manejar multiples peticiones concurrentes eficientemente.

MongoDB

MongoDB fue seleccionado por su flexibilidad para almacenar datos semi-estructurados sin necesidad de definir esquemas rigidos. Esto es particularmente util para:

  • Almacenar contenido variable de paginas web (CMS)
  • Guardar metadatos de imagenes de diferentes formatos
  • Facilitar iteraciones rapidas durante el desarrollo
  • Excelente rendimiento en operaciones de lectura (importante para cargar contenido web)

Docker

Docker garantiza que el sistema funcione de forma identica en desarrollo y produccion, eliminando el clasico problema "funciona en mi maquina". Los contenedores Docker encapsulan todas las dependencias necesarias, facilitando el deployment y reduciendo conflictos de versiones.

Railway (Plataforma Cloud)

Railway fue seleccionado por:

  • Simplicidad: No requiere conocimientos avanzados de DevOps o infraestructura cloud
  • Deployment automatico desde Git con un solo click
  • Certificados SSL gratuitos y automaticos (HTTPS)
  • Facturacion por uso real (pay-as-you-go), ideal para proyectos academicos
  • Soporte nativo para Docker y multiples lenguajes
  • Logs centralizados y monitoreo incluido

Arquitectura de Microservicios

Aunque este es un proyecto academico, la arquitectura de microservicios refleja las mejores practicas de la industria moderna (utilizada por empresas como Netflix, Amazon, Uber). Esta decision arquitectonica:

  • Demuestra comprension de patrones de diseño de software profesional
  • Facilita futuras expansiones del sistema (añadir mas modelos, nuevas funcionalidades)
  • Permite trabajar en componentes de forma independiente
  • Es escalable para posibles implementaciones clinicas reales

8. Flujo Completo del Sistema

A continuacion se describe el flujo completo desde que un usuario accede al sistema hasta que recibe el resultado del analisis:

Paso 1: Acceso a la Aplicacion Web

El usuario accede a la URL del sistema desde su navegador web. Railway envia la aplicacion React al navegador del usuario, donde se renderiza la interfaz completa.

Paso 2: Carga de Imagen

El usuario selecciona una imagen de fondo de ojo desde su dispositivo. La imagen es validada en el frontend (formato, tamaño) antes de ser enviada.

Paso 3: Transmision al Backend

La imagen se envia desde React al backend FastAPI mediante una peticion HTTP POST. La imagen se transmite en formato multipart/form-data.

Paso 4: Validacion en Backend

FastAPI valida el tipo de archivo, tamaño y formato de la imagen. Si la validacion falla, se retorna un mensaje de error al usuario.

Paso 5: Envio al Modelo de IA

El backend envia la imagen al microservicio de IA mediante una peticion HTTP interna. El modelo recibe la imagen y comienza el procesamiento.

Paso 6: Procesamiento por el Modelo

El modelo de Deep Learning preprocesa la imagen (redimensionamiento, normalizacion), la pasa a traves de su red neuronal y genera una prediccion con el nivel de severidad y el porcentaje de confianza.

Paso 7: Retorno de Resultados

El modelo retorna los resultados al backend, que los formatea y agrega recomendaciones clinicas basadas en la severidad detectada.

Paso 8: Almacenamiento (Opcional)

El backend puede guardar en MongoDB un registro del analisis (sin datos personales identificables) para analisis estadistico posterior.

Paso 9: Visualizacion de Resultados

El frontend recibe el resultado y lo muestra al usuario de forma clara, incluyendo el diagnostico, el nivel de confianza, y las recomendaciones medicas pertinentes.

9. Consideraciones Tecnicas Adicionales

9.1 Seguridad

  • Todas las comunicaciones se realizan a traves de HTTPS (SSL/TLS)
  • Validacion estricta de tipos de archivo para prevenir ataques de inyeccion
  • Limites de tamaño de archivo para prevenir ataques de denegacion de servicio
  • Las imagenes procesadas no se almacenan permanentemente para proteger privacidad

9.2 Rendimiento

  • Frontend compilado y minificado para carga rapida
  • Lazy loading de componentes React para reducir tiempo inicial de carga
  • Compresion de imagenes antes de transmision
  • Cache de recursos estaticos en el navegador
  • Procesamiento asincrono para evitar bloqueo del servidor

9.3 Escalabilidad

  • Arquitectura diseñada para escalar horizontalmente (añadir mas instancias)
  • Base de datos MongoDB puede configurarse con replicas para alta disponibilidad
  • Posibilidad de implementar balanceador de carga para multiples instancias del modelo
  • Cola de procesamiento para manejar picos de trafico

10. Resultado

El sistema RetinaVision AI representa una implementacion moderna y escalable de una aplicacion web medica que integra inteligencia artificial. La arquitectura seleccionada combina tecnologias probadas en la industria con practicas de diseño que garantizan mantenibilidad, escalabilidad y rendimiento.

La separacion en microservicios permite que cada componente del sistema pueda evolucionar independientemente, facilitando futuras mejoras del modelo de IA, expansion de funcionalidades o integracion con sistemas hospitalarios existentes.

La seleccion de tecnologias open-source (React, FastAPI, MongoDB) asegura que el proyecto pueda ser mantenido y mejorado a largo plazo sin dependencias de software propietario, alineandose con los principios de investigacion academica abierta y reproducible.
